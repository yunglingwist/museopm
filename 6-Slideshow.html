Project Charter — Overview

What Is This Project?

Cilekler is a browser-based interactive storytelling game commissioned by a museum institution to increase visitor interest and attract new audiences to an upcoming exhibition. The game follows a Time Detective narrative theme, requested by the museum to match the exhibition's subject matter. Players move through branching story choices that reveal historically grounded clues, and every path ends with an invitation to visit the museum in person.

The story is authored in Twine and delivered as a standalone HTML/CSS website that runs in any standard browser, built with original AI-generated images and sound assets. The project is produced by a four-person academic team within the Master's programme in Digital Humanities and Digital Knowledge (DHDK) at the University of Bologna.

Project Charter — Scope & Stakeholders

Scope & Stakeholders

Stakeholders

Main stakeholder: the commissioning museum institution.
Other stakeholders: prospective and on-site museum visitors, exhibition curators, museum communication and digital staff, academic supervisors.

In Scope

  • A complete, playable Twine story with multiple decision points and at least two endings
  • A styled HTML/CSS website built from the Twine export, deployable on GitHub Pages
  • Original AI-generated images and sound assets supporting atmosphere and immersion
  • Story content thematically aligned with the museum's upcoming exhibition
  • Access via QR code, usable on the museum's website and on-site at the exhibition
  • Project management documentation covering all phases from initiation to delivery

Out of Scope

  • Back-end infrastructure, databases, or user account systems
  • Native mobile app development
  • Full multilingual localisation (an Italian version is a possible stretch goal)
  • Real-time analytics or visitor data collection

Project Charter — Constraints & Risk Analysis

Constraints & Risk Analysis

Constraints

  • Team size & timeline: four students working within one academic semester
  • Toolchain: authored in Twine; final product must remain a static HTML/CSS site with no server-side dependencies
  • Content approval: narrative referencing the exhibition requires museum sign-off, which may cause delays
  • Asset licensing: every image and audio asset must be original or free to use
  • No user data: the game must not collect, store, or transmit personal data

Risk Register

RiskProb.ImpactMitigation
Narrative scope creep;story branches grow too large to completeHighHighFix a maximum number of passages and endings from the outset; enforce limits via the Twine story map
Content approval delays from the museumMediumHighUse placeholder content during development; submit for review early
Inconsistent visual/audio asset qualityMediumMediumAgree a style guide and generation parameters before production; review assets as a team
HTML conversion breaks Twine logic or stylingMediumHighTest the converted website at each integration step; keep a clean backup of the Twine source
Coordination issues across a four-person remote teamMediumMediumUse GitHub for version control; define clear handoffs between writing, Twine and HTML phases
Game unplayable on some browsers or devicesLowHighTest on Chrome, Firefox and Safari, desktop and mobile, before delivery

Software Requirements Specification — Features & Domain Model

Features & Domain Model

Product Features

  • Branching narrative with multiple choices and at least two distinct endings
  • Time Detective theme, with clues and characters tied to the exhibition
  • Original AI-generated visual assets for atmosphere and historical context
  • Ambient sound design to enhance immersion
  • Browser-based delivery on desktop and mobile, no installation
  • Exhibition call to action at the conclusion of the story
  • QR code accessibility for on-site deployment
  • Short completion time of about 10-15 minutes

Domain Model

  • Story : the overall game: title, theme, audience, duration, list of passages
  • Passage : one narrative unit, with text, image/audio reference and choices
  • Choice : an option the player selects, linking to the next passage
  • Asset : an image or audio file used within a passage
  • Ending : a terminal passage that closes a path, with a call to action
  • PlayerSession : the path taken in the current browser session; stored in-session only, never collected

System Architecture & Application Diagram

System & Application Architecture

System Architecture

Developer machines and the Twine authoring tool feed a network layer (GitHub repository, GitHub Pages, the HTML/CSS bundle and the asset package). The browser requests these files over HTTP; the museum visitor reaches the game either by opening the URL directly or by scanning a QR code provided by the museum institution.

Application Diagram

Inside the browser, the UI layer (Title Screen, Passage, Choice, Ending and Asset Display) is driven by an application layer that loads passages, handles player choices and tracks the session. These components read from a data layer made of the Twine story content, the stylesheet, the asset bundle and the browser's session storage.

Sample Use Case

Prospective Visitor Discovers the Exhibition

Primary Actor
Prospective museum visitor (teenager, adult, or family member)
Secondary Actors
Museum institution; web browser; GitHub Pages
Goal
The visitor plays the Cilekler Time Detective game online, engages with the museum's exhibition theme, and is motivated to visit the physical exhibition.
Trigger
The visitor encounters the game link via the museum's website, social media, or a QR code at the entrance, and opens it in a browser.

Basic Flow

  1. The visitor opens the game link from the museum's website, a QR code, or social media. No installation or account required.
  2. The Title Screen shows the game's title and the Time Detective premise; the visitor clicks "Start Investigation."
  3. The Story Engine loads the first passage, with an atmospheric image and ambient audio setting the scene.
  4. The visitor reads each passage and selects a choice; the Choice Handler registers it and the next passage loads.
  5. At a key decision point the story diverges significantly, one path toward solving the mystery, another toward a misleading outcome.
  6. The visitor reaches an Ending passage: a narrative conclusion plus a call to action inviting a museum visit.
  7. The visitor can choose "Play Again" to try a different path, or simply close the browser. No personal data is collected at any point.

Alternative flows: if an asset fails to load, the story text remains readable and play continues; if the visitor closes and reopens the browser, the game returns to the Title Screen for a fresh, independent playthrough.

Testing Methodology & Test Case

Testing Methodology

Objectives

Verification : confirm the game matches the SRS: branches lead to valid endings, assets load correctly, and the call to action is present.

Validation : confirm the game fits the museum context: engaging, historically appropriate, completable in 10–15 minutes, and effective in motivating exhibition visits.

Test Levels

  • Narrative unit testing (each passage)
  • Asset integration testing
  • Branch & path testing (system / end-to-end)
  • Cross-browser & responsive testing
  • Performance testing (load time)
  • Domain validation (content review)
  • Privacy & data handling check

Sample Test Case

"Player Completes a Full Narrative Path to an Ending"

Type: System / End-to-End, manual

Pass ifEvery passage loads, no broken links or errors occur, and the ending shows a visible call to action.
Fail ifA blank page or script error appears, a choice leads to a broken link, a dead end is reached that is not a designated ending, the ending lacks a call to action, or load time exceeds 5 seconds.

Release Readiness

  • All Mandatory requirements implemented & covered by a passing test
  • Playable to an ending in Chrome, Firefox and Safari
  • No Critical or Major defects remain open
  • Call to action present in every ending; game publicly accessible via GitHub Pages

User Manual — Highlights

User Manual Highlights

Game Screens

ScreenWhat you seeWhat you do
Title ScreenGame title, Time Detective intro text, "Start Investigation" buttonRead the premise and click "Start Investigation"
Story PassageNarrative text, an atmospheric image, optional ambient audio, 2–3 choicesRead the text, then click a choice to continue
Choice PointTwo or more clickable choice buttons, each leading to a different pathClick the choice that reflects your instinct
Ending ScreenConclusion of the story, a summary, and a call to actionRead the ending, then click the museum link or "Play Again"

How to Access

Option A Browser: open the URL; the Title Screen loads automatically, no download required.

Option B QR code: scan the code at the museum entrance and tap the link to open the game on a mobile browser.

Privacy

No personal data is collected, stored, or transmitted. No account is required. The game uses no cookies, analytics, or external tracking.

Troubleshooting

  • Game does not load: check the connection, refresh, or try another browser
  • Images do not appear: refresh once; story text stays readable regardless
  • No sound: check device volume; sound is optional and not required to play
  • A choice button does not respond: tap again, or refresh to restart
